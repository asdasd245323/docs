---
title: "Intent: completePorting"
description: "How do I let a user complete number porting?"
---

Connect will open on the complete porting page for a subscription. If the user has previously started the porting process, Connect will resume from where the user left off. Upon completing all required porting steps, users will be redirected back to `callbackUrl` if provided.

![completePorting.jpg](https://i.gigscdn.net/docs/v1/connect-session-completePorting.jpg)

The request expects a `completePorting` object that defines the `subscription` for which to complete a porting.

### Creating a Connect Session with completePorting intent

```bash
curl --request "POST" \
  --url "/api/projects/${GIGS_PROJECT}/connectSessions" \
  --header "Content-Type: application/json" \
  --header "Authorization: Bearer ${GIGS_TOKEN}" \
  --data '{
    "callbackUrl": "https://example.com",
    "intent": {
      "type": "completePorting",
      "completePorting": {
        "subscription": "sub_0U32VAxJ0Z7P8Y40mBAnOP9LPhEr"
      }
    },
    "user": "usr_0U2ViuFW0Z7P8Y2S1L76mfADMT6m"
  }'
```

<Note>
Please confirm that the subscription has a porting with status `declined` or `informationRequired` associated with it before creating a Connect Session. 
</Note>

## Intent Behavior

By default, the `completePorting` intent displays an **intro page** at the start and an **outro page** after completion. Use the `behavior` object to control which screens are shown.

| Parameter | Description |
| --- | --- |
| `hideIntroPage` | When `true`, skips the intro page and navigates users directly to the first required step. |
| `hideOutroPage` | When `true`, skips the outro page and returns users immediately to the `callbackUrl`. |

### Create a Connect Session with hidden intro and outro pages

```bash
curl --request "POST" \
  --url "/api/projects/${GIGS_PROJECT}/connectSessions" \
  --header "Content-Type: application/json" \
  --header "Authorization: Bearer ${GIGS_TOKEN}" \
  --data '{
    "callbackUrl": "https://example.com",
    "intent": {
      "type": "completePorting",
      "completePorting": {
        "subscription": "sub_0U32VAxJ0Z7P8Y40mBAnOP9LPhEr",
        "behavior": {
          "hideIntroPage": true,
          "hideOutroPage": true
        }
      }
    },
    "user": "usr_0U2ViuFW0Z7P8Y2S1L76mfADMT6m"
  }'
```

## Prefill Required Porting Information

By default, the `completePorting` intent collects all required information. If your application already knows this data, you can prefill it by [creating a User](/api/gig-core-api/users/create-a-user) or [creating an Address](/api/gig-core-api/user-addresses/create-a-user-address).

Alternatively, when [creating the Porting](/api/gig-core-api/portings/create), you can pass this information directly:

```bash
curl --request "POST" \
  --url "/api/projects/${GIGS_PROJECT}/portings" \
  --header "Content-Type: application/json" \
  --header "Authorization: Bearer ${GIGS_TOKEN}" \
  --data '{
    "phoneNumber": "+19591234567",
    "provider": "p4",
    "firstName": "Jerry",
    "lastName": "Seinfeld",
    "birthday": "2017-07-21",
    "address": {
      "city": "New York City",
      "country": "US",
      "line1": "129 West 81st Street",
      "postalCode": "10024",
      "state": "NY"
    },
    "user": "usr_0SNlurA049MEWV4OpCwsNyC9Kn2d"
  }'
```

## Retrying Failed Porting Attempts

The `completePorting` intent can also be called for previously failed porting requests. This feature applies to specific `declinedCode`s such as `portingAccountNumberRequiredOrInvalid` or `portingAccountPinRequiredOrInvalid`.

## Aborting the Porting Flow

Users can abort the porting flow. When this happens, they are redirected to the `callbackUrl` with the status `user_aborted`.
